58    /**
59    * This operator searches sequential patterns in a set of transactions. Each transaction must be
60    * encoded as an single example and must contain one attribute for the time and for the customer.
61    * This pair of attribute is used for generate one sequence per customer containing each single
62    * transaction ordered by the time of each transaction. The algorithm then searches sequential
63    * patterns in the form of: If a customer bought a and c in one transaction, he bought b in the
64    * next: <a, c> then <b>. The minimal support describes how many customer must support such a
65    * pattern for regarding it as frequent. Infrequent patterns will be dropped. A customer supports
66    * such a pattern, if there are some parts of his sequence including the pattern. The above pattern
67    * would be supported by a customer with this transactions: <s, g> then <a, s, c> then <b> then <f,
68    * h>.
69    *
70    * The parameters min_gap, max_gap and window_size determine how transaction are handled. For
71    * example, if the above customer forgot to by c, and had to return 5 minutes later to buy c, then
72    * his transactions would look like that: <s, g> then <a, s> then <c> then <b> then <f, h> This
73    * would not support the pattern <a, c> then <b>. To avoid this problem, the window size determines,
74    * how long a subsequent transaction is treated as the same transaction. If the window size is
75    * larger than 5 minutes, the <c> would be treated as being part of the second transaction and hence
76    * this customer would support the above pattern.
77    *
78    * The max_gap parameter causes a customers sequence not to support a pattern, if the transactions
79    * containing this pattern are to widely separated in time. The min_gap parameter does the same if
80    * they are to near.
81    *
82    * @author Sebastian Land
83    */
