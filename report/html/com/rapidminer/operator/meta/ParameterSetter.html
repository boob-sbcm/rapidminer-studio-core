45    /**
46    * Sets a set of parameters. These parameters can either be generated by a
47    * {@link ParameterOptimizationOperator} or read by a
48    * {@link com.rapidminer.extension.legacy.operator.io.ParameterSetLoader}. This operator is useful, e.g. in the
49    * following scenario. If one wants to find the best parameters for a certain learning scheme, one
50    * usually is also interested in the model generated with this parameters. While the first is easily
51    * possible using a {@link ParameterOptimizationOperator}, the latter is not possible because the
52    * {@link ParameterOptimizationOperator} does not return the IOObjects produced within, but only a
53    * parameter set. This is, because the parameter optimization operator knows nothing about models,
54    * but only about the performance vectors produced within. Producing performance vectors does not
55    * necessarily require a model. <br/>
56    * To solve this problem, one can use a <code>ParameterSetter</code>. Usually, a process with a
57    * <code>ParameterSetter</code> contains at least two operators of the same type, typically a
58    * learner. One learner may be an inner operator of the {@link ParameterOptimizationOperator} and
59    * may be named &quot;Learner&quot;, whereas a second learner of the same type named
60    * &quot;OptimalLearner&quot; follows the parameter optimization and should use the optimal
61    * parameter set found by the optimization. In order to make the <code>ParameterSetter</code> set
62    * the optimal parameters of the right operator, one must specify its name. Therefore, the parameter
63    * list <var>name_map</var> was introduced. Each parameter in this list maps the name of an operator
64    * that was used during optimization (in our case this is &quot;Learner&quot;) to an operator that
65    * should now use these parameters (in our case this is &quot;OptimalLearner&quot;).
66    *
67    * @author Simon Fischer, Ingo Mierswa
68    */
