42    /**
43    * TableSorter is a decorator for TableModels; adding sorting functionality to a supplied
44    * TableModel. TableSorter does not store or copy the data in its TableModel; instead it maintains a
45    * map from the row indexes of the view to the row indexes of the model. As requests are made of the
46    * sorter (like getValueAt(row, col)) they are passed to the underlying model after the row numbers
47    * have been translated via the internal mapping array. This way, the TableSorter appears to hold
48    * another copy of the table with the rows in a different order.
49    * <p/>
50    * TableSorter registers itself as a listener to the underlying model, just as the JTable itself
51    * would. Events recieved from the model are examined, sometimes manipulated (typically widened),
52    * and then passed on to the TableSorter's listeners (typically the JTable). If a change to the
53    * model has invalidated the order of TableSorter's rows, a note of this is made and the sorter will
54    * resort the rows the next time a value is requested.
55    * <p/>
56    * When the tableHeader property is set, either by using the setTableHeader() method or the two
57    * argument constructor, the table header may be used as a complete UI for TableSorter. The default
58    * renderer of the tableHeader is decorated with a renderer that indicates the sorting status of
59    * each column. In addition, a mouse listener is installed with the following behavior:
60    * <ul>
61    * <li>
62    * Mouse-click: Clears the sorting status of all other columns and advances the sorting status of
63    * that column through three values: {NOT_SORTED, ASCENDING, DESCENDING} (then back to NOT_SORTED
64    * again).
65    * <li>
66    * SHIFT-mouse-click: Clears the sorting status of all other columns and cycles the sorting status
67    * of the column through the same three values, in the opposite order: {NOT_SORTED, DESCENDING,
68    * ASCENDING}.
69    * <li>
70    * CONTROL-mouse-click and CONTROL-SHIFT-mouse-click: as above except that the changes to the column
71    * do not cancel the statuses of columns that are already sortoing - giving a way to initiate a
72    * compound sort.
73    * </ul>
74    * <p/>
75    * This is a long overdue rewrite of a class of the same name that first appeared in the swing table
76    * demos in 1997.
77    *
78    * @author Philip Milne
79    * @author Brendon McLean
80    * @author Dan van Enckevort
81    * @author Parwinder Sekhon
82    * @author Ingo Mierswa
83    *
84    * (only for Java 5 generics parts and some other small enhancements)
85    *
86    */

384    public void tableChanged(TableModelEvent e) {
385    // If we're not sorting by anything, just pass the event along.
386    if (!isSorting()) {
387    clearSortingState();
388    fireTableChanged(e);
389    return;
390    }
391   
392    // If the table structure has changed, cancel the sorting; the
393    // sorting columns may have been either moved or deleted from
394    // the model.
395    if (e.getFirstRow() == TableModelEvent.HEADER_ROW) {
396    cancelSorting();
397    fireTableChanged(e);
398    return;
399    }
400   
401    // We can map a cell event through to the view without widening
402    // when the following conditions apply:
403    //
404    // a) all the changes are on one row (e.getFirstRow() == e.getLastRow()) and,
405    // b) all the changes are in one column (column != TableModelEvent.ALL_COLUMNS) and,
406    // c) we are not sorting on that column (getSortingStatus(column) == NOT_SORTED) and,
407    // d) a reverse lookup will not trigger a sort (modelToView != null)
408    //
409    // Note: INSERT and DELETE events fail this test as they have column == ALL_COLUMNS.
410    //
411    // The last check, for (modelToView != null) is to see if modelToView
412    // is already allocated. If we don't do this check; sorting can become
413    // a performance bottleneck for applications where cells
414    // change rapidly in different parts of the table. If cells
415    // change alternately in the sorting column and then outside of
416    // it this class can end up re-sorting on alternate cell updates -
417    // which can be a performance problem for large tables. The last
